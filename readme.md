# Курс по теории языков программирования

### Архитектура и зона ответственности Weasel VM

**Роль и философия**
Weasel VM — это строго типизированная, стековая виртуальная машина, выступающая в роли одного из бэкендов для системного языка программирования Weasel. Главная философия VM: «Абсолютное доверие фронтенду». Виртуальная машина не занимается проверкой типов в рантайме, разрешением имен, поиском методов или неявным приведением (кастами). Все эти высокоуровневые концепции разрешаются на этапе семантического анализа (Sema). VM получает на вход максимально низкоуровневый, "разжеванный" и типизированный байткод (Chunk), который остается только детерминированно и быстро исполнить.

### Стек и система типов

В отличие от классических интерпретаторов динамических языков, стек Weasel VM не использует тяжеловесные обертки (например, std::variant или Tagged Unions с проверкой тегов). Стек представляет собой плоский массив сырых 64-битных машинных слов. Поскольку компилятор уже доказал типобезопасность программы, VM использует специализированные инструкции (например, OP_ADD_SINGLE, OP_BIT_AND_U_LITTLE), которые точно знают, как интерпретировать биты на вершине стека.

Сложные математические типы (векторы, матрицы) не существуют для VM как единые высокоуровневые сущности. Компилятор осуществляет процесс "распаковки" (Flattening), превращая операции над векторами в последовательности атомарных скалярных инструкций.

### Управление пулом констант (Constant Pool)

Сам байткод (Chunk) содержит статический пул констант, в котором хранятся литералы программы: числа, строки и другие неизменяемые данные.

Инструкции вида `OP_CONSTANT`, `OP_LOAD_STRING` и аналогичные не создают значения напрямую, а загружают на стек индекс или адрес элемента из этого пула. Это позволяет эффективно работать с данными, превышающими размер одного машинного слова (например, строки или крупные числовые литералы), не нарушая модель 64-битного стека.

### Управление памятью (ARC)

VM не использует фоновый трассирующий сборщик мусора (Mark-and-Sweep GC). Управление динамической памятью (кучей) для структур и сложных объектов построено на механизме автоматического подсчета ссылок (Automatic Reference Counting — ARC). Компилятор Weasel самостоятельно рассчитывает время жизни объектов и вставляет в байткод атомарные инструкции `OP_RETAIN` и `OP_RELEASE`.

Задача VM — просто инкрементировать или декрементировать счетчик ссылок по команде и освобождать память (используя RAII и семантику умных указателей C++), когда счетчик достигает нуля.

### Поддержка массивов и буферов

Объекты в куче представлены не только фиксированными структурами, но и динамическими массивами переменной длины (slices) и сырыми буферами (buffer).

Такие структуры представляют собой составные объекты (например: указатель + длина + вместимость) и управляются через тот же механизм ARC.

Доступ к элементам массива осуществляется через индексированные инструкции (например, `OP_LOAD_INDEX`, `OP_STORE_INDEX`), которые работают с адресом памяти и смещением, уже вычисленным компилятором.

### Структуры данных и вызовы

* Weasel VM ничего не знает об именах полей структур или названиях методов. Парадигма UFCS (Uniform Function Call Syntax) полностью раскрывается фронтендом до генерации байткода.
* Для работы со структурами VM использует инструкции `OP_ALLOCATE_STRUCT`, `OP_GET_FIELD` и `OP_SET_FIELD`, которые оперируют исключительно статическими целочисленными смещениями (байтовыми оффсетами), заранее рассчитанными компилятором.
* Динамическая диспетчеризация отсутствует, любые вызовы (включая методы) превращаются в обычные инструкции `OP_CALL` по фиксированным адресам с использованием фреймов вызова (CallFrames) для изоляции локальных переменных.

### Интеграция с внешним миром (FFI)

Weasel VM полностью изолирована от операционной системы и внешнего мира. Взаимодействие осуществляется исключительно через механизм нативных функций (Foreign Function Interface).

Специальные инструкции (`OP_CALL_NATIVE`) передают управление заранее зарегистрированным C++ функциям-оберткам. Эти функции:

* извлекают аргументы со стека VM
* выполняют необходимое действие (ввод-вывод, работа с файлами, графикой и т.д.)
* возвращают результат обратно в стек

Таким образом, вся небезопасная и платформозависимая логика строго вынесена за пределы VM.

### Обработка ошибок (Runtime Traps vs UB)

Ошибки в Weasel VM разделяются на две принципиально разные категории:

**1. Runtime Traps (штатные ловушки выполнения)**
Это корректно обрабатываемые ошибки времени выполнения:

* деление на ноль
* переполнение стека вызовов
* явный вызов `panic()`

В таких случаях VM останавливает выполнение и формирует диагностический отчёт (например, через `DiagnosticData`) с трейсом вызовов.

**2. Internal Compiler Errors (некорректный байткод)**
Это ошибки, свидетельствующие о баге компилятора:

* стек опустился ниже нуля
* неизвестный или повреждённый опкод
* нарушение контрактов инструкций

В этих случаях VM завершает работу немедленно (Undefined Behavior), так как предполагается, что корректный байткод **никогда** не должен вызывать такие ситуации.

### Что VM не делает

* Не строит и не обходит AST: Эта задача строго закреплена за фронтендом (парсер и семантический анализатор).
* Не хранит строковые имена: Переменные и поля структур не ищутся по строковым ключам в рантайме (нет хеш-таблиц для scope). Всё заменено на индексы стека и смещения памяти.
* Не собирает мусор в фоне: Отсутствуют фазы остановки мира (Stop-the-World) для поиска потерянных объектов.
* Не прощает ошибок компилятора: Некорректный байткод приводит к немедленному падению, сигнализируя об Internal Compiler Error.
